# 介绍

一个美术人员使用的 Git 桌面端工具，名称叫 Artistic Git。

- 它基于 Tauri + React + shadcn ui + lucide icon + Vite + Vitest，全部技术栈均使用最新稳定版本；前端包管理器使用 pnpm。
- 支持自动更新，运行时每小时检查更新，并有主动检查更新按钮（位于设置界面“关于”分区）。
  - 自动检查发现新版本：不弹窗打断，静默后台下载；下载完成后弹出更新提示框：新版本号 + 更新日志（来自 Release notes）+ “立即重启更新”/“稍后”（稍后则下次启动时自动安装）。
  - 手动检查：立即显示“发现新版本”弹窗与下载进度，其余同上；检查失败提示“检查失败”（自动检查失败则静默忽略，下小时重试）。
  - 安全约束：有 git 操作正在执行、或存在未完成的冲突解决/审查模式时，“立即重启更新”按钮禁用并提示原因，绝不在操作中途重启。
  - 多窗口：更新提示只在一个窗口弹出——弹在最近获得焦点的窗口，该窗口关闭则转移到下一个最近焦点窗口；重启安装会关闭所有窗口，任一窗口有进行中操作都阻止。
- 支持 GitHub Actions 编译构建，并支持手动触发和推送 main 自动触发；发布仅允许来源为 main 且仓库变量 `ENABLE_MAIN_RELEASE=true`，不使用人工审批，未开启闸门时只执行测试与 dry-run 打包。
- 版本号自动递增，自动上传二进制文件到 Github Releases，自动收集提交信息作为更新日志。
- 需要编译所有平台的产物。
- 对于核心逻辑需要编写严格的单元测试和端到端测试（不关心测试时长，用真实临时项目测试，而不是 fake/mock 命令），该工具涉及到资产丢失问题，需要首要保证安全性和稳定性。
- 当代码发生非预期错误时禁止静默处理。
- 记录详细错误日志，发生错误时弹出专门的错误详情弹窗，并支持复制错误信息和打开日志文件所在目录方便发送给开发人员。
- 如果错误会影响到工具之后的运行（例如状态异常），必须弹出专门的奔溃详情弹窗，提供按钮让用户重启工具。
- 界面响应式设计，支持任意窗口尺寸下最佳显示，默认尺寸 1280x720。

# 设计总则

- 设计上应简单人性化，美术人员对 Git 技术并不了解，在使用过程中应尽量避免存在对 Git 概念和操作理解的要求，将主要工作流设计为可视化操作，在工具内部自动执行 Git 命令。
- 界面应简洁精美，有苹果和 Linear 的设计感，多使用图标以增加亲近感和友好性。
- 合理使用符合直觉、苹果设计风格的动画效果。
- 超出长度的文本使用简略显示，如："/sdfsdf/sdfsd/fsdf..."， 鼠标悬停弹出 tooltip 显示完整路径。
- 文件路径显示全平台统一使用 `/` 分隔符（与 git 内部一致，Windows 上不混用 `\`）。
- 保持界面元素自身文案的简练性，需要更多说明则使用悬停 tooltip 展示。
- 支持同时打开多个窗口管理多个项目：
  - 单进程多窗口（Tauri 原生多 WebView 窗口），每个窗口独立承载一个项目。
  - 打开新窗口的唯一入口：菜单栏 File → New Window / `Cmd/Ctrl+N`，新窗口显示起始界面；界面内不提供打开新窗口入口。
  - 同一项目不允许重复打开：再次打开已打开的项目时，直接聚焦到已有窗口（发起打开的起始窗口保留不关闭），从根上消除并发写锁（.git/index.lock）风险。
  - 应用层面单实例（second-instance 转发给已运行进程），避免双进程绕过上述保护。
  - 关闭最后一个窗口即退出应用（Windows/Linux）；macOS 遵循平台惯例，Dock 保活。
  - 关窗保护：关闭窗口（点 X / `Cmd/Ctrl+W`）时，若该窗口有写操作进行中、或存在未完成的冲突解决/审查模式，弹确认框“有操作正在进行，关闭将取消该操作并恢复到操作前状态”；确认则执行标准取消/恢复流程后再关闭。退出应用（`Cmd/Ctrl+Q`）逐窗口执行同样检查。
  - 崩溃隔离：避免一个窗口崩溃拖垮全部窗口。
    - 渲染层：各平台 WebView 本身为独立渲染进程，单窗口 WebView 崩溃不影响其他窗口，主进程检测到后自动重载该窗口并弹出崩溃详情弹窗。
    - 前端层：每个窗口独立的 React 错误边界与全局错误处理器，JS 异常仅影响自身窗口。
    - 主进程层（Rust）：所有命令入口 Result 化，禁止 panic 外泄；panic hook 记录日志并转为崩溃弹窗上报，不直接 abort 进程；git 操作均在独立子进程中执行（`std::process::Command` 调用内嵌 git，非 Tauri sidecar 机制，见“技术架构与实现约定”），单个任务失败不影响主进程。
- 支持中英文双语切换，自动根据系统语言切换，支持手动切换语言。
- 支持深色和浅色主题切换，自动根据系统主题切换，支持手动切换主题。
- 所有破坏性操作均需要弹出二次确认提示框。

# 独特设计

- 工具内嵌 Git，所有 Git 操作均使用内嵌 Git 执行。
  - 实现方式：把整套 git 与 git-lfs 发行目录作为 Tauri `resources`（打包资源）分发，运行时经 `resource_dir()` 解析路径，用 Rust `std::process::Command` 直接调用（**不使用 Tauri `externalBin`/sidecar 单文件机制**，因为 git 是一整套发行目录而非单个二进制）；由一个薄的 “git runner” 封装层统一承载飞行锁、进度解析、日志与 `Result<T, AppError>` 化。
  - 二进制来源（按平台）：Windows 使用官方 **MinGit** 预编译包；macOS/Linux 从**官方源码**在 CI 以 `RUNTIME_PREFIX` 可重定位方式自建——macOS 用 Xcode 工具链出 arm64 + x86_64 再 lipo 成 Universal；Linux 在 **Ubuntu 20.04 容器（glibc 2.31）**编译，依赖库（libcurl/openssl/zlib/pcre2/expat）全部静态链入，`NO_GETTEXT/NO_TCLTK` 裁剪并保留 Perl-backed porcelain（如 `git submodule`），glibc 本身动态链接（不采用 musl 全静态，规避其 DNS/NSS 行为差异）；git-lfs 全平台用官方预编译包（macOS 合成 Universal）。
  - 内嵌 Git LFS：同时打包 git-lfs 二进制，打开或克隆的仓库若包含 LFS 规则（.gitattributes），自动执行 `git lfs install --local`，对用户完全透明；该命令允许按需持久写入仓库级 `.git/config` 的 `filter.lfs.*`，属于工具主动写入白名单。
  - 版本锁定：git 与 git-lfs 版本钉入**单一构建配置文件**（版本号 + 各平台产物/源码 SHA-256 + 完整编译配方集中于此），不随构建机环境浮动；实现启动时取当时最新稳定版钉入，升级 = 修改该文件走 PR + 全量测试并记录在更新日志，保证全平台全用户跑同一套经过测试的 git 行为。git 版本优先选 fsmonitor 支持面最全的版本，个别平台缺失则自动降级（见“实时 Fetch”的本地侧配套）。内嵌 Git/LFS/SSH 二进制产物不提交到普通 Git 仓库；CI 通过 cache/artifact 复用，本地开发通过脚本准备 dev resources。
  - 构建期校验（按平台细化）：预编译产物（Windows MinGit、全平台 git-lfs）校验发布页**二进制 SHA-256**；macOS/Linux 自建的 git 则钉死**官方源码 tarball 的 SHA-256 + 完整编译配方（flags/容器镜像）**以可复现。防止供应链污染或下载损坏。
  - 运行期自检：启动时执行 `git --version` / `git lfs version`，确认可执行且版本符合预期；不符或无法执行则致命错误崩溃弹窗。
  - 隔离系统 git：所有调用显式使用内嵌路径，隔离 `GIT_EXEC_PATH`/`PATH`，绝不回退到用户系统 git；git-lfs 通过内嵌 git 的 filter 配置指向内嵌 lfs。
  - 配置隔离：调用时用受控环境（`GIT_CONFIG_NOSYSTEM=1` 忽略系统级 config、`HOME` 指向受控目录，实际 Git 命令不读取用户全局 Git config）。向导预填与身份懒校验需要全局身份时，由工具通过专用只读路径显式读取真实 `~/.gitconfig` 的 `user.name`/`user.email`，不让用户全局配置影响核心 Git 命令行为。
- 远程仓库地址：工具只管理一个远程仓库地址，名称为 `origin`；即使仓库存在多个 remote，也只对 `origin` 执行显示、Fetch、同步、推送和项目设置修改。
  - 若存在 `origin` 以外的 remote，允许打开并在项目设置或打开提示中说明“仅管理 origin，其它远程不会被操作”。
  - 若没有 `origin` 但存在其它 remote，仍按**无远程模式**处理，并提示用户可在项目设置中配置 origin。
  - 远程地址可以为空（未配置 origin 或在项目设置中清空）：此时仓库进入**无远程模式**——隐藏所有同步入口（项目级/分支级同步按钮、待同步 Badge、提交/撤回确认框中的“立即推送”复选框），提交/撤回跳过前置同步，全部功能离线可用；主界面顶部显示“未配置远程仓库”警告条，引导去项目设置填写。
  - 有远程但某分支无上游（仅本地分支）：该分支的提交/撤回跳过前置同步；“立即推送”勾选含义 = `git push -u origin <分支>` 首次发布该分支；分支级同步按钮 = 发布分支（拉取侧为空集，与“同步始终双向”自洽）；项目级批量同步不自动发布仅本地分支（避免误发布私人实验分支）。
- 认证：同时支持 SSH 和 HTTPS 两种远程认证方式，克隆输入框同时接受两种地址，不做转换。
  - SSH：host key 校验使用 `accept-new` 策略（首次连接自动信任，后续指纹变更则报错），避免交互式确认卡死流程。
    - ssh 二进制来源（按平台）：Windows 捆绑**钉死版本的 Win32-OpenSSH 官方发行包**（MinGit 不含 ssh；随 resources 分发并校验 SHA-256，不依赖系统自带 OpenSSH 以免版本浮动）；Win32-OpenSSH 版本选择规则为**优先官方 stable `OpenSSH-Win64.zip`，若官方没有 stable Win64 包则允许钉死最新官方 preview 包作为显式 fallback，且一旦 stable 出现 CI gate 必须失败并要求切回 stable**；macOS 13+ / Linux 用**系统 ssh**（这两平台自带现代 OpenSSH，且密钥本就放 `~/.ssh` 与系统生态互通）。统一通过 `-c core.sshCommand="<ssh 路径> -o StrictHostKeyChecking=accept-new -o UserKnownHostsFile=~/.ssh/known_hosts"` 注入，不落盘。
    - 密码短语（passphrase）：注入极小的 `SSH_ASKPASS` 帮助程序 + `SSH_ASKPASS_REQUIRE=force`，ssh 需要密码短语时经本地 IPC 回调主进程弹出输入框，彻底不依赖 TTY、永不卡死；用户首次输入后主进程在内存缓存，后续操作（含定时 Fetch）静默返回、不再打扰；不自主管理 ssh-agent（若系统已把密钥加载进 agent，ssh 优先用 agent、askpass 不触发）。
    - 后台绝不弹框：定时 Fetch 等后台只读操作标记为 `non-interactive`，需要密码短语而内存无缓存时直接归入“可预期认证/网络失败”走离线图标，仅用户主动发起的操作才允许 askpass 弹框。
    - 跨会话记忆（可选）：提供“记住密码短语”开关，勾选后存入系统钥匙串（与 HTTPS 凭据同一套存储）；默认关闭。
  - HTTPS：凭据形式为用户名 + Personal Access Token，默认按 host 维度共享（如 github.com、gitlab.example.com 各一份），同时支持 `protocol + host + path` 的覆盖项，以兼容同一企业 Git host 下不同组织/路径需要不同 Token 的场景。
    - 存储位置：系统钥匙串（macOS Keychain / Windows Credential Manager / Linux Secret Service），绝不明文落盘。
    - 实现机制：内置 credential helper 模式，每条 git 命令注入 `-c credential.helper=<本工具>`，git 需要凭据时回调工具（走 `get/store/erase` 原生协议），401 重试与凭据填充由 git 原生流程驱动。
    - 本地 IPC 底座（凭据 helper 与 SSH askpass helper 共用）：主进程监听 **Unix domain socket（macOS/Linux）/ 命名管道（Windows）**（`interprocess` crate 抽象），**不开网络端口**；高层用户操作使用 `operation-id` 贯穿恢复与 UI 上下文，每条 git 子命令再派生独立 `invocation-id + 一次性 token`，环境变量注入 `socket 路径 + token + invocation-id`（不落盘），helper 连上校验后 token 立即失效（Unix socket 权限 0600、Windows 管道 ACL 限本用户）。交互性决策收敛主进程：由 `operation-id/invocation-id → {是否交互, 仓库, host, path}` 表决定“弹框 / 静默返回缓存 / 直接失败”，后台 Fetch 的 invocation 恒为非交互。两个 helper 均为本工具自建的单文件二进制，随 `resources` 分发。
    - 首次操作 HTTPS 仓库且无凭据时，弹出凭据输入框（host 只读、用户名、Token 密码框），附小字说明可在 Git 平台的 Access Tokens 页面生成，输入后自动继续原操作。
    - 认证失败（401）时弹出同一凭据框，提示 Token 无效或已过期。
    - 用户取消输入则视为该操作失败，走常规错误恢复流程。
- 内部会实时 Fetch 远程分支：
  - 定时执行 `git fetch origin --prune`（prune 保证远程已删除的分支同步消失），间隔可在设置中配置，默认 60 秒。
  - 额外触发时机：打开项目时、窗口重新获得焦点时、执行同步/提交/撤回等操作前。
  - 并发控制：全局 git 操作单飞行（single-flight），若有用户操作正在执行则跳过本轮定时 Fetch，避免锁冲突。
  - 失败表现：后台 Fetch 的网络失败属于可预期错误，不弹错误弹窗，改为在项目信息区域显示离线小图标 + tooltip（显示最后成功 Fetch 时间），网络恢复后自动消失；非网络类意外错误仍走错误弹窗流程。
  - 本地侧配套（巨型二进制仓库友好）：变更检测的重活交给内嵌 git 的 **fsmonitor**——通过 `-c core.fsmonitor=true -c core.untrackedCache=true` 让 `git status` 在几万~几十万文件的大树上近乎瞬时（只动 `.git` 本地内部：index 扩展 + 本地守护进程，不碰共享内容、同事无感）。fsmonitor 为**平台条件优化**：macOS/Windows 必开；Linux 若所钉 git 版本不支持内建 fsmonitor daemon，则退化为仅 `core.untrackedCache=true` + 自有 watcher（大仓库依然可用）。我们自己的文件系统 watcher 分层启用：`.git` 关键路径（HEAD/refs/index/packed-refs/MERGE_HEAD 等）必须精监听以驱动分支与中间态状态；工作区 watcher 仅作为触发刷新信号，尝试启用、可静默降级，真正状态仍以防抖后的 `git status` 为准。
  - 自产事件抑制：我们自己的 git 操作持写锁期间忽略 watcher 事件（已知是自身造成），操作结束后统一刷新一次，消除 LFS 检出等自产事件风暴。
  - 防抖 + 单飞：工作区事件合并约 300–500ms，同一时刻最多一个 `git status`（有写锁或已有一个在跑则跳过）；界面保持实时。
  - 超限静默降级：极端大树触达 OS watch 上限时静默退化为“git fsmonitor + 粗粒度轮询”，不打扰用户。
- 分支：显示分支时将本地分支和远程分支合并显示，远程分支名称会去掉前缀 `origin/`，因为并不考虑存在多个远程仓库的情况。
  - 合并规则：同名的本地分支与远程分支合并为一个列表项。三种存在状态：仅本地（未推送过）→ 分支图标；本地 + 远程 → 分支图标（云端状态用 Badge 表达）；仅远程（本地没有）→ 云分支图标。
  - 切换到“仅远程”分支：自动创建本地跟踪分支（`git checkout -b X origin/X`），对用户透明，图标随之变为分支图标。
  - 删除“仅远程”分支：确认框中“是否删除远程分支”复选框强制勾选且禁用，文案明确显示将删除远程分支。
  - 排序：当前分支置顶，其余按最近提交时间降序。
  - 仅远程分支不显示待同步 Badge（无本地 ahead/behind 概念）。
- 本地更改：不区分暂存区和工作区，修改、新增、删除，所有未提交的更改都统一称为本地更改。
  - 严格遵循 .gitignore：被忽略的文件不出现在本地更改列表（与 git status 一致）；未跟踪且未被忽略的文件正常显示为“新增”，可勾选提交/储藏/还原。
  - 所有丢弃类操作统一走系统回收站兜底：先把工作区当前版本按仓库相对路径收集到临时目录（如 `Artistic Git discarded changes/<repo>/<timestamp>/path/to/file`）并移入系统回收站，再恢复目标 Git 状态；新增/未跟踪文件表现为删除进回收站，修改文件表现为当前版本进回收站后 checkout 恢复。UI 仍使用红色二次确认并提示“此操作无法撤销”，不承诺回收站一定可恢复。
- 文件规范化与二进制判定：
  - 换行符：不修改 `core.autocrlf`，尊重用户/系统既有的 Git 配置。
  - 不主动改写用户的 `.gitattributes`（属于仓库内容，由技术负责人管理，工具无权代改）。
  - 二进制判定统一使用 git 自身的机制（git diff 的 binary 标记 / null 字节探测），Diff 与冲突界面据此决定走文本还是二进制展示，不依赖扩展名白名单。
  - “整仓库显示已修改”自愈（可选增强，默认不打扰）：若检测到本地更改数量异常巨大（疑似换行符/权限位/filemode 问题），在本地更改列表顶部显示提示条，建议一键规范化（内部 `git add --renormalize` 预览，不自动提交）。
- 仓库健康检查：打开项目时做一次健康检查，识别异常状态并引导：
  - 游离 HEAD（detached HEAD）：提示“当前未处于任何分支上”，提供“新建分支保存当前状态”或“切换到某分支”的引导。
  - 残留中间态（外部工具留下的 rebase/merge/cherry-pick 未完成）：检测 .git 下中间态标记，提示“检测到未完成的 Git 操作”，提供“放弃并恢复”（abort）按钮。
  - 空仓库（无任何提交，unborn HEAD）：不做引导。分支列表显示 unborn 的当前分支名（如 `main`）+ 当前标记，历史选项卡显示“尚无提交”空态；本地更改与提交照常可用（首次提交后一切正常化）；此状态下切换/删除/新建分支入口禁用（unborn HEAD 下语义不全）。
  - `.git/index.lock` 残留（上次异常退出）：**永不自动清除**。因“同一项目仅一个窗口”，打开时本工具必然未在对其跑 git，故该锁只可能是崩溃残留或外部进程正在使用；弹**引导式提示**（非致命）显示锁文件存在时长（mtime 年龄）并警示“若你正在别处对本仓库操作请取消，否则可安全清除”，由用户**显式确认**后才删除（不做不可靠的跨平台进程归属探测）。
  - 无法自动修复的检查结果归入致命错误走崩溃弹窗；可自动修复的走引导提示。
- 子模块支持（安全优先，对用户尽量透明）：
  - 实现分两档、同版本交付：**Tier 0 消费型**（子模块作为锁定到指定提交的只读依赖被检出，覆盖“引用共享库但不改它”的绝大多数场景）；**Tier 1 编辑型**（在子模块内改文件：聚合更改视图 + 归属徽章 + 自底向上提交 + 安全护栏）。含子模块的仓库不半支持就是坏的，故两档都在 v1 落地。
  - 克隆/打开：克隆用 `git clone --recurse-submodules`；打开若存在 `.gitmodules` 则自动 `git submodule update --init --recursive`（持写锁、显进度、含子模块内 LFS）。
  - 定时 Fetch 用 `--recurse-submodules=on-demand`，保证子模块对象/LFS 可用。
  - 同步：超级项目 ff 拉取后自动 `git submodule update --init --recursive`，把各子模块检出到新锁定的提交；消费型用户全程无感。
  - 本地更改统一视图：聚合超级项目和各子模块工作区的更改进同一列表，每个子模块的文件用小徽章标注归属（如 `子模块: assets-lib`）；超级项目的指针变化显示为友好行“子模块 X 更新到新版本”，而非裸 gitlink diff。
  - 提交自底向上：先做完所有本地工作（在受影响子模块检出跟踪分支并带过更改、子模块提交、超级项目指针提交）并校验，**最后才按依赖顺序 push（子模块 → 超级项目）**；整体作为一个带备份的多步骤融合操作，用户只看到一次“提交”。其回滚遵从下文“发布边界”：本地阶段可整体回滚，一旦某次 push 成功则不再倒退——**子模块已推、超级项目 push 失败时，超级项目那条指针提交作为普通“未推送提交”留在本地、下次同步补推（前向安全的部分完成，而非回滚）**。
  - 安全护栏：子模块默认游离 HEAD，用户要在子模块提交更改时先自动检出其跟踪分支（`.gitmodules` 的 branch 或默认分支）并把更改带过去，无法安全进行则明确阻止并说明；冲突界面文件路径带子模块前缀，复用同一套 Diff/冲突组件。
- 创建分支：创建时，需用户输入分支名称和基于的分支，默认基于当前所在分支；并且复选框：是否远程分支，默认勾选；是否立即切换到该分支，默认勾选。
  - “是否远程分支”勾选的含义 = 创建本地分支后立即 `git push -u origin <名称>` 在远程建立同名分支；不勾选 = 仅本地。
  - 分支名实时校验：非法字符/格式（git 规则）、与现有分支重名时，输入框下方红字提示且确认按钮禁用。
- 分支选择通用规则：所有需要选择分支的地方（基于的分支、自动跟踪设置等）均可选择“仅远程”分支；若该操作要求本地分支存在，则自动创建对应的本地跟踪分支。
- 删除分支：删除时，需用户选择是否删除远程分支，默认不勾选。
  - 当前所在分支不可删除：删除按钮禁用 + tooltip “无法删除当前所在的分支”。
  - 未合并提交保护：若待删分支包含未被任何其他分支包含的提交，确认框中红字警告“该分支包含 N 个未合并的提交，删除后将丢失”。
  - 删除远程分支 = `git push origin --delete <名称>`；若因权限被拒（受保护分支），按常规错误弹窗处理，本地删除结果保留。
  - 默认分支（origin/HEAD 指向者）不做额外删除保护：现有保护已足够（当前分支不可删 + 未合并提交红字警告 + 远程受保护分支由平台侧拒绝）。
- 同步：默认情况下，同步操作会根据设置中是否开启自动跟踪来决定具体流程。
  - 同步语义：同步始终是双向操作——先拉取远程新提交，再推送本地未推送的提交。下文“拉取 X 分支”均指：远程有新提交时以 ff-only 方式合并到本地对应分支，绝不自动产生 merge commit，保持线性历史。
  - 同步入口作用范围：
    - 项目级同步按钮（项目信息区域）= 串行批量同步所有“有对应远程分支的本地分支”，按分支列表既有排序执行（当前分支最先，其余按最近提交时间降序）；每个分支按其情形选用对应流程（当前分支走完整流程，非当前分支走快路径引用更新/慢路径 worktree）。
    - 分支级同步按钮（列表项 hover）= 仅同步该指定分支。
    - 批量同步中某分支需交互处理（冲突解决/远程历史改写决策）时就地暂停处理，完成或取消后继续余下分支；全部处理完后弹出汇总（逐分支成功/失败/需关注）。单个分支的不可重试失败不中断整个批量，记录后继续。
  - 待同步提交数 Badge = ahead + behind 之和，tooltip 中分开显示（如 “↑2 待推送 ↓3 待拉取”）；项目信息区域同步按钮变橙色的条件 = 任一“有对应远程分支的本地分支”存在 ahead > 0 或 behind > 0。
  - 非当前分支的同步：对非当前检出的分支执行同步时，不切换分支、不碰当前工作区：
    - 快路径：无分叉时用引用更新完成拉取（`git fetch origin B:B`，仅 fast-forward），再推送；无需储藏（非当前分支不存在本地更改）。
    - 慢路径：分叉需要 rebase、或自动跟踪需要合并时，自动在应用 cache/data 下的固定前缀临时目录创建工具专用 worktree（`git worktree add`）检出该分支，在 worktree 中执行完整同步流程，完成后删除 worktree。
    - worktree 中的冲突同样进入冲突解决界面（文件路径指向 worktree），取消则 abort 并删除 worktree。
    - 清理保证：操作结束（成功/失败/取消）均删除工具专用 worktree；启动时只自动清理工具创建的临时 worktree 及其管理记录，用户自己的 linked worktree 仍拒绝打开且不自动清理。
  - 推送竞态自愈：push 因 non-fast-forward 被拒（fetch 与 push 之间同事又推了新提交）时，自动重新走一遍“fetch → ff/rebase → push”，最多循环 3 次；仍失败则提示“远程更新过于频繁，请稍后重试”并恢复到操作前状态；全程显示“正在同步…”，无感重试。
  - 绝不使用 force push：任何常规同步流程都不含 `--force`；push 为普通 push（仅 fast-forward 成功），任何非快进一律触发上面的自愈循环，杜绝覆盖他人提交。唯一改写远程的场景是“远程历史被改写”防护中用户显式选择“以远程为准”（本地 reset，非 force push 到远程）。
  - 无内容可同步：ahead=0 且 behind=0 时操作瞬间完成，同步按钮短暂显示“已是最新”（对勾 + 淡出），不弹框；批量同步全部无需同步则汇总提示“所有分支均已是最新”。
  - 分叉处理：本地与远程分叉（ff-only 失败）时，自动 rebase 本地未推送的提交到远程最新提交之上，维持线性历史（对未推送提交 rebase 是安全的，不改写共享历史）。
    - rebase 冲突时弹出冲突解决界面（“他人的更改”=远程内容，“自己的更改”=本地提交内容），用户取消则 `git rebase --abort` 完整恢复到操作前状态。
    - 全程对用户透明，界面上不出现 rebase 概念，仅显示“正在同步”。
    - 远程历史被改写防护（他人 force push）：判定条件为“本地已推送过的提交不在远程历史中”，此时不自动 rebase，弹出专门提示框说明远程历史发生变化，提供选项：1) 以远程为准——本地独有提交先自动创建备份分支 `backup/<分支名>-<时间戳>`，再硬重置到远程（默认推荐项）；2) 取消。若原分支名包含 `/`，备份分支保留原分支层级。
  - 自动跟踪：可以设置 A 分支追踪 B 分支，设置了自动追踪的 A 分支在执行同步操作时会使用不同的流程。
  - 开启自动跟踪时：
    - 如果 A 分支有本地更改，则自动储藏。
    - 拉取 A 和 B 分支。
    - 自动 ff only 合并 B 分支到 A 分支，如果失败则提示，然后进入恢复储藏流程。
    - 推送 A 分支到远程仓库。
    - 进入恢复储藏流程。
  - 关闭自动跟踪时：
    - 如果 A 分支有本地更改，则自动储藏。
    - 拉取 A 分支。
    - 推送 A 分支到远程仓库。
    - 进入恢复储藏流程。
- 储藏：如果是软件内部自动储藏，则名称以 “Auto Stash:” 开头，并说明理由，并且整个流程成功后删除储藏，否则保留储藏。
- 恢复储藏：
  - 如果能够成功恢复储藏，则无需任何额外操作。
  - 如果储藏的更改和当前分支有冲突，则弹出冲突解决界面。
  - 手动 `git stash apply` 在执行前必须建立可恢复点；若应用后进入冲突界面且用户取消，工作区、索引与 HEAD 恢复到 apply 前状态，原储藏始终保留。
- 切换分支：
  - 如果本地有修改，切换分支时弹出确认框，让用户选择如何处理：
    1. 自动储藏并恢复本地更改。（默认）
    2. 丢弃本地更改（同样先把工作区当前版本移入系统回收站兜底）。
  - 选项 1 的语义为“更改跟人走”：储藏 → 切换到目标分支 → 在目标分支立即恢复储藏（同一流程内完成），适配“改错分支了”的高频场景。
  - 恢复时若冲突则进入冲突解决界面；流程成功后自动删除该 Auto Stash，失败则保留。
  - 文件列表勾选状态在切换前后保持。
- 提交更改：
  - 提交范围 = 本地更改列表中勾选的文件（默认全部勾选）。内部实现：`git add -A -- <勾选路径>` 后 `git commit`，未勾选的文件保持为本地更改不受影响，新增/修改/删除文件统一处理。
  - 点击提交按钮（无勾选文件时禁用）弹出二次确认框，包含：提交信息输入框（多行，placeholder 提示描述本次修改内容，为空时确认按钮禁用）、将提交的文件数、复选框“是否立即推送到远程仓库”（默认勾选；无远程模式下隐藏；当前分支无上游时勾选含义为 `push -u` 首次发布）。
  - 大文件体检（提交执行前，非阻断强提醒）：扫描将提交的文件，命中“大小超过阈值（默认 50MB，可在项目设置调整）且未被 .gitattributes 的 LFS 规则覆盖”的文件时，弹出“检测到大文件”对话框，列出文件及大小，提供三选：1) 将这些文件类型加入 LFS 并继续（默认推荐，自动 `git lfs track`、更新 .gitattributes、转为 LFS 指针后提交，需明示将修改 .gitattributes）；2) 仍以普通文件提交（确认知情）；3) 取消提交。
    - 例外：仓库未初始化 LFS（无 .gitattributes 也无 LFS 规则）时仅弹一次说明，尊重用户选择，不强推。
    - “是否启用大文件检查”开关与阈值放设置的项目设置区。
  - 提交签名：工具不主动改写签名配置；若提交因 `commit.gpgsign=true` 但无可用签名密钥而失败，弹出提示框解释原因，让用户选择“为本仓库关闭提交签名并继续”（写入仓库级 `commit.gpgsign=false`/`tag.gpgsign=false`）或取消。
  - 确认后自动先同步，直到成功再进行下一步骤，否则中断（当前分支存在上游时提交因此**需要联网**；无上游/无远程则跳过前置同步、可离线提交；见“隐私与网络行为”的联网要求）。
    - 同步会储藏全部本地更改（含未勾选文件），同步成功恢复储藏后按勾选范围提交；恢复储藏若冲突则中断提交流程进入冲突解决界面，文件勾选状态全程保持。
  - 执行提交，如果因为未同步等可通过重试的错误失败，则再次自动同步。
- 撤回提交：
  - 底层语义：统一使用 `git revert`（生成反向提交抵消目标提交），永不改写共享历史，已推送/未推送提交行为一致。
  - 可撤回当前分支上的任意提交（不限于最新一条）；入口在提交详情面板中，按钮标签“撤回此提交”。
  - merge commit 的撤回按钮禁用（tooltip 说明“合并提交无法撤回”）。
  - 撤回产生的提交信息自动生成，固定英文：`Revert: <原提交信息>`（不随 UI 语言变化，见国际化章节）。
  - revert 遇到冲突时进入冲突解决界面，用户取消则 `git revert --abort` 恢复原状。
  - 撤回的二次确认框中有复选框，是否立即推送到远程仓库，默认勾选（无远程模式下隐藏；无上游时勾选含义为 `push -u` 首次发布）。
  - 仅能够对当前分支的提交进行撤回，否则按钮禁用并用 tooltip 提示要切换到对应分支。
  - 撤回前会自动同步，直到成功再进行下一步骤，否则中断（当前分支存在上游时撤回因此**需要联网**；无上游/无远程则跳过前置同步、可离线撤回；见“隐私与网络行为”的联网要求）。
  - 撤回前自动储藏本地更改。
  - 执行撤回，如果因为未同步等可通过重试的错误失败，则再次自动同步。
- 审查模式：
  - 在主界面点击 “审查模式” 按钮，进入审查模式，此时覆盖层阻止所有操作，中间有一个 “退出审查模式” 按钮。
  - 审查模式即会先将本地修改自动储藏（`Auto Stash: review mode`），然后**仅拉取（pull-only，只拉不推）**以让工作区反映最新远程内容——审查是“看”的模式，绝不把未推送的本地提交作为进入审查的副作用发布出去。
  - 进入不做联网前置检查：流程固定为“储藏 → 尝试拉取”。拉取因网络失败按可预期错误处理（离线小图标 + tooltip，不弹窗、不回滚进入），用户基于本地最新提交照常审查，网络恢复后由“有新内容需同步”提示接管。无远程模式下跳过拉取步骤，语义 = 检查工作区是否有漏提交内容（审查模式入口不隐藏）。
  - 覆盖层内容：除“退出审查模式”按钮外，显示当前分支名与最新提交信息，让用户确认自己在审查什么。
  - 期间行为：定时 Fetch 继续；远程有新提交时不自动拉取，在覆盖层上提示“有新内容需同步”并显示同步按钮，点击后执行拉取（此时工作区干净，ff 拉取无风险），完成后更新显示的提交信息。
  - 退出审查模式时会自动恢复储藏的本地修改；若冲突则进入冲突解决界面；成功恢复后删除该 Auto Stash。
  - 崩溃保护：进入审查模式时在项目级数据中记录标记；若应用崩溃/被强杀后重新打开该项目，检测到未正常退出的审查模式标记且对应 Auto Stash 存在，弹窗提示“上次审查模式未正常退出，是否恢复当时储藏的本地更改？”，防止用户更改被遗忘在储藏里。
- 以上所有自动多步骤融合操作首先要确保操作有备份，并遵循**发布边界原则**（因“绝不 force push”，已成功推送到共享远程的提交无法安全撤回）：把每个融合操作切成“**本地阶段（可逆）**”与“**发布阶段（不可逆）**”。为压缩不可逆窗口，先做完所有本地工作（储藏、检出、提交、指针更新）并校验，最后才按依赖顺序 push。
  - “备份”的具体矩阵（不为每个操作建可见备份）：同步中的 rebase 只动未推送提交，靠 `rebase --abort` + reflog 恢复，不建备份分支；“远程历史改写、以远程为准”场景建 `backup/*` 分支；工作区更改靠失败保留的 Auto Stash 或丢弃前的系统回收站兜底；revert/提交为追加式操作，无需备份。
  - 本地阶段任一步不可重试地失败 → 完整回滚到操作前状态。
  - 发布阶段失败 → **绝不 force 撤销已成功的 push**；先走网络类自愈重试（最多 3 次指数退避），仍失败则停在一个“**前向安全**”的终态（本地提交完整保留为未推送、下次同步补推）并明确报告，绝不留下损坏/游离状态。
  - 无论停在哪，工作区与 HEAD 都必须干净可用。
- 操作并发模型：
  - 单写锁 + 忙时拒绝（非真队列）：每个窗口维护一把“仓库写锁”，同步、提交、撤回、切换、创建/删除分支、储藏、审查模式进出等写操作**互斥串行**，同一时刻只有一个在跑；第二个写操作在忙碌时**直接禁用/拒绝、绝不入队**（前文“写操作队列”一词即指此互斥语义，不是 FIFO 队列）。杜绝 index.lock 冲突和状态互踩。
  - 认证弹框是操作的延续、非新操作：写操作进行中 git 回调凭据 helper / askpass 时，对应的凭据框/密码短语框**必须能在忙碌态照常弹出**（它是当前操作的 continuation，不受“忙时禁用写入口”约束）；服务 helper 回调的 IPC 任务与持写锁的操作任务是并发两条线（持锁任务等 git、git 等 helper、helper 等弹框），架构上必须避免互相阻塞导致死锁；在认证框上取消 = 取消该操作、走恢复流程。
  - 忙碌态 UI：有写操作进行时顶部显示细进度条 + 当前操作名（如“正在同步…”），其他写操作入口禁用（灰显 + tooltip “有操作正在进行”）；只读操作不受影响（浏览历史、看 Diff、搜索照常）。
  - 外部并发：本工具靠“同一项目仅一个窗口”消除自身并发；用户在终端/其他 GUI 同时操作同一仓库无法阻止，依赖 git 自身的 `index.lock` 报错 + 健康检查兜底，不额外加锁。
  - 长操作可取消：克隆、同步、审查模式进入等耗时操作显示取消按钮，取消走标准的“恢复到操作前状态”流程。
  - 进度来源：能拿到进度的（clone/fetch/push/LFS 传输，解析 git `--progress` stderr）显示百分比；拿不到的显示不确定进度条（spinner）。
  - 定时 Fetch 属于后台只读操作，持有写锁时跳过（single-flight），不阻塞用户。
- 非目标（v1）：不提供任意分支间的合并操作（含 merge commit 语义）；分支间合流仅存在于自动跟踪同步的 ff-only 合并。完整的合并功能列入后续版本规划（冲突解决等基础设施已可复用）。

# 数据存储

- 全局设置（语言、主题、Fetch 间隔、Git 用户信息）：Tauri `appConfigDir/settings.json`。
- 项目级数据（自动跟踪规则、侧栏比例、最近打开时间）：`appDataDir/projects.json`，按规范化绝对路径为 key；绝不写入仓库内部，避免影响同事。
- 最近项目列表：由 projects.json 派生（按最近打开时间排序）。
- HTTPS 凭据：系统钥匙串（已在认证设计中定义），默认按 host 共享，并允许 `protocol + host + path` 级覆盖。
- 日志：`appLogDir/`，按天滚动文件，保留 30 天；结构化文本，含时间戳、级别、执行的 git 命令与完整输出（实现用 `tracing` + 按天滚动 appender）。
- 钥匙串访问：HTTPS 凭据与（可选的）SSH 密码短语经 `keyring` crate 存取系统钥匙串。
- 窗口尺寸/位置：**按项目存**（放入 `projects.json`，每个项目窗口恢复自己上次的大小/位置，契合多窗口多项目）；仅“起始/新建窗口”用一份全局默认几何。
- 配置并发（单进程多窗口）：`settings.json` 与 `projects.json` 是多窗口共享文件，所有读写走**唯一的 Rust 侧配置 actor**——内存唯一模型 + mutex 串行写 + 写临时文件 rename 原子落盘 + 高频更新（如拖分隔条比例）防抖；projects.json 走“读改写结构化模型”只改动对应项目条目，杜绝盲覆盖丢更新。全局设置变更后除落盘外向**所有窗口**广播事件即时生效。
- Git 用户信息：工具配置为唯一数据源，打开/克隆项目时以工具身份为准写入仓库级 `.git/config`（user.name/user.email）——**仅当当前仓库值与之不同才实际写（含覆盖仓库既有的不同身份），只动 `.git/config`、绝不改全局 `~/.gitconfig`**；若工具尚无身份（跳过了向导）则不写，交由“身份懒校验”在首次需要时补齐。向导首次运行与身份懒校验需要全局 fallback 时，工具通过专用只读逻辑读取真实 `~/.gitconfig` 的身份字段，而不是让 Git Runner 在受控 `HOME` 外读取全局配置。在全局设置中修改身份后，广播所有窗口并立即对每个已打开仓库执行同样的“不同才写”逻辑，消除“设置已改、当前项目仍用旧身份”的窗口期。
- SSH Key：使用标准位置 `~/.ssh/`（检测 id_ed25519/id_rsa 等常见名称），生成时默认 ed25519 类型存到 `~/.ssh/id_ed25519`，与系统生态互通。
- 工具主动写入的仓库级 `.git/config` 项（清单）：
  - 自动（打开/克隆时）：`user.name`/`user.email`（身份）；`filter.lfs.*`（`git lfs install --local`）。
  - 用户显式发起：`remote.origin.url`（项目设置修改远程地址）；`commit.gpgsign`/`tag.gpgsign`（仅签名导致提交失败、用户确认后写入 false）。
  - 命令级注入不持久化：`credential.helper`、`core.sshCommand`（accept-new）、`core.longpaths=true`（仅 Windows，深路径美术仓库的高频翻车点，应用打包同时启用 long path aware）、rename detection、`--progress` 等仅通过 `-c` 传入，不写入 config。

# 内部安全产物的生命周期

- 备份分支 `backup/*`：分支列表默认隐藏（安全网，非工作分支）；命名为 `backup/<分支名>-<时间戳>`，若原分支名含 `/` 则保留层级；提供“查看安全备份”入口列出所有 `backup/*`（含原分支与时间，解析失败时显示完整 ref），用户确认无用后手动删除；纯本地永不推送；不自动删除。
- 失败保留的自动储藏 `Auto Stash:`：正常显示在储藏区，详情标注“由 <某操作> 自动创建”说明来历；不自动删除。
- 启动时孤儿清理仅限明确安全的工具临时产物：只清理应用 cache/data 下固定前缀的工具专用 worktree 及其 Git 管理记录；用户自己的 linked worktree 不自动清理。`index.lock` 不自动清除，改由健康检查以引导式二次确认交用户处置（见“仓库健康检查”）；`backup/*` 分支与储藏也不在自动清理之列。
- 原则：在“不丢数据”和“不堆积”之间取安全侧——自动产物可见、可手动清理，但工具绝不自动删除可能有价值的数据。

# 错误处理

- 三级错误分类：
  - 可预期错误：网络失败、认证失败（401）、合并/储藏冲突、用户取消、输入校验失败。各自有专门 UI（离线图标、凭据框、冲突解决界面等），不弹通用错误弹窗。
  - 非预期错误：git 命令未知失败、IO 错误、内部逻辑异常，但不影响后续运行。弹出错误详情弹窗：用户语言的错误摘要 + 可折叠技术详情（git 命令、退出码、完整 stderr）+ 按钮：复制错误信息、打开日志目录。
  - 致命错误：影响后续运行的状态异常（仓库残留 rebase/merge 中间态且无法自动 abort、内部状态机损坏、Rust panic）。弹出崩溃详情弹窗：同错误详情内容 + “重启工具”按钮，覆盖层阻止其他操作。
- 自动重试策略（多步骤流程中的网络类失败）：最多 3 次，指数退避（1s → 2s → 4s）；3 次后视为“无法通过重试解决”，提示并恢复到操作前状态。
- 实现要求：所有 Rust 命令返回 `Result<T, AppError>`，`AppError` 携带分类、操作上下文、git 命令与输出；每个错误在弹窗出现前已完整写入日志。

# 技术架构与实现约定

- 契约先行：跨轨道共享契约以 Rust 类型为真相源，覆盖 Tauri command 签名、`AppError` JSON、事件 payload、resources 目录布局、Diff/冲突数据结构与前端 props。TypeScript bindings 由 Rust 类型自动生成（如 `serde` + `specta`/`tauri-specta` 或等价方案），CI 重新生成并检查 diff 必须为空，禁止手写类型长期漂移。
- 状态归属：**Rust 后端是唯一真相源**，前端是响应式视图。后端持有内嵌 git runner、写锁/操作队列、`.git` watcher、定时 Fetch 调度、仓库状态缓存。
  - 两条通道：命令（请求/响应，查询与动作，一律 `Result<T, AppError>`）+ 事件（后端向指定窗口推送 `repo-changed`/`operation-progress`/`fetch-state`/`conflict-entered` 等）。
  - 前端状态一分为二：**TanStack Query** 管服务端态（按“仓库 + 查询类型”为 key，watcher 事件到达即定向 `invalidateQueries`，git log 分页用 infinite query）；**Zustand** 管 UI 态（选中的提交、勾选集合、面板开合、视图模式、分隔条比例）。
  - 按窗口隔离：每个 WebviewWindow 是独立 React 根、独立 store 与 query client，只绑定一个项目；后端按 window label 路由事件。
- 文本 Diff 与冲突编辑引擎：统一用 **CodeMirror 6** 一个引擎同时承载“diff 展示（`@codemirror/merge`，分栏/行内，语法高亮映射到深/浅色令牌）”和“冲突手动编辑”；图片/其他二进制/LFS 指针走独立自定义组件（棋盘格图片对比、文件卡片）。文件分类、重命名检测、二进制/LFS 判定、大文件保护由 Rust 侧 git 权威决定，编辑器只负责渲染与编辑。
- 提交历史图表：**泳道（lane）布局在 Rust 侧按 200 一页增量计算**（`git log --parents` topo-order，跨页把打开的泳道状态带到下一页保证连续，输出每行的泳道线段/节点/颜色）；前端用 **TanStack Virtual** 做行虚拟化 + 轻量 **SVG** 绘制预计算好的线段。不使用现成 gitgraph 库（不满足万级虚拟化与极简美学）。
- 内容搜索（`git log -S` pickaxe）：防抖 + 可取消 + 搜索期间 loading，避免大仓库卡顿。
- LFS 文件锁定（`git lfs lock/unlock`）：纳入设计蓝图，**v1 仅做架构预留**——文件列表项、提交前检查、`.gitattributes` 处理均预留“锁状态”接入位；完整的锁定/解锁工作流、锁状态展示、管理员强制解锁留待后续阶段（前提平台支持 LFS 锁定 API，如 GitHub/GitLab）。

# 测试策略

- 核心 Git 流程测试（Rust 集成测试，主体）：每个用例在临时目录创建真实 bare 仓库作为远程 + 一到两个真实 clone（第二个 clone 模拟“同事”制造分叉/冲突），用内嵌 git 执行全真实命令。测试 harness 只从 `ARTISTIC_GIT_DIST_DIR` 或打包 resources 指向的显式内嵌 Git 目录取可执行文件；缺失或版本不匹配即失败，绝不回退系统 git。
  - 覆盖：同步、自动跟踪同步、分叉 rebase、储藏/恢复、冲突解决、撤回、审查模式、以及每个流程的失败恢复路径（在任意步骤注入失败，断言恢复到操作前状态）。
  - LFS 场景同样真实跑。
- 认证栈测试：本地起 `git http-backend` + Basic Auth 模拟 HTTPS 401/凭据流程；SSH 流程用本地 sshd 容器（仅 Linux CI 跑）。
- 前端单元测试：Vitest + Testing Library（组件逻辑、状态管理）。
- E2E：`tauri-driver` + WebdriverIO，驱动真实应用操作真实临时仓库，跑通“克隆→修改→提交→同事推送→同步→冲突→解决→撤回”全链路。
  - 现实约束：tauri-driver 不支持 macOS（WKWebView 无 WebDriver），E2E 在 Linux + Windows CI 上跑，macOS 由 Rust 集成测试兜底（核心 git 逻辑与平台无关）。
- CI：GitHub Actions——单元 + 集成测试跑全部三平台矩阵，E2E 跑 Linux/Windows；CI 通过 git-dist cache/artifact 向测试提供 `ARTISTIC_GIT_DIST_DIR`。

# 构建与发布

- 应用标识：产品名 `Artistic Git`，可执行文件名 `artistic-git`，bundle identifier `com.smallmain.artistic-git`。
- 平台产物：
  - macOS：Universal 二进制（Intel + Apple Silicon 合一），`.dmg`（首次安装）+ `.app.tar.gz` + 签名（更新包）。
  - Windows：x64，NSIS 安装包（`.exe`），采用 **perUser（currentUser）安装**——自更新无需 UAC 提权、可静默完成。
  - Linux：x86_64，`.AppImage`（主推产物，自带 webkit、兼容 glibc 2.31+）+ `.deb`（要求 Ubuntu 22.04+ / webkit2gtk-4.1）。
  - 最低支持版本（README 明示）：macOS 13+（Universal）；Windows 10 1809+（x64，WebView2 门槛）；Linux 见上。
- 自动更新：Tauri updater，以 GitHub Releases 为更新源（latest.json），更新包用 Tauri 签名密钥对签名（私钥存 GitHub Secrets）；更新完整性与 OS 签名无关。仓库与 Releases 必须公开（updater 免认证直接访问 latest.json 的前提）。
  - macOS 前提（App Translocation）：未签名应用从“隔离”状态启动会被放到只读随机路径运行、导致自更新失败；因此**引导用户把应用移动到 /Applications 并右键→打开一次**（清除隔离后原地自更新才正常）。
  - Linux 非对称：**AppImage 支持应用内自动更新**；**`.deb` 不自更新**（归系统包管理器），其“检查更新”降级为提示 + 打开 Release 页面手动升级。
- 代码签名：暂无 Apple Developer 账号与 Windows 代码签名证书，不做 OS 级签名（Tauri 更新签名照做，更新安全不受影响）；README 说明 macOS 首次打开需右键→打开、Windows SmartScreen 的绕过步骤；后续有证书再补。
- 版本号策略：初始版本 `0.1.0`（`1.0.0` 留给功能完备里程碑）；基于约定式提交（Conventional Commits）自动判定递增级别——仅 fix → patch；含 feat / refactor 等 → minor；含破坏性变更（BREAKING CHANGE / `!`）→ major；无法从提交解析出级别时默认 patch。
  - 推送到 main 自动构建发布：解析自上一 tag 以来的提交信息，计算版本号并打 tag；该流程仅在来源为 main 且仓库变量 `ENABLE_MAIN_RELEASE=true` 时发布，不使用人工审批，闸门未开启时只执行测试与 dry-run 打包，不创建公开 Release、tag 或 `latest.json`。
  - 手动触发（workflow_dispatch）默认同样自动计算，亦可手动指定版本级别覆盖。
  - 更新日志 = 自上一个 tag 以来的所有提交信息。
  - PR 只跑测试，不发布。

# 起始界面

- 打开该工具判断是否首次进入，是则打开向导界面，否则打开起始界面。
  - 首次进入判定：以全局 settings.json 中的 `onboarded: true` 标志为准（不存在或 false 即视为首次）；向导完成或点“跳过”后置为 true。判定依据应用自身状态，不依赖 git 配置是否存在。
- 左侧是两个带图标的文字按钮，克隆项目、打开项目。
  - 克隆项目弹窗：仓库地址（SSH/HTTPS 均可）+ 目标父目录（目录选择器，记住上次使用的父目录）+ 目录名（自动从 URL 推断，可修改）。
    - 克隆过程：进度条弹窗（解析 `git clone --progress` 的百分比，LFS 下载阶段单独显示进度），支持取消——取消后自动清理半成品目录（二次确认）。
    - 成功后自动打开该项目进入主界面，并加入最近项目列表。
  - 打开项目：目录选择器。所选目录不是 Git 仓库时提示“该目录不是有效的 Git 项目”，不提供 `git init`。
    - 选中仓库子目录时自动解析到仓库根（`git rev-parse --show-toplevel`）并以根路径打开/记录，静默纠正不弹确认。
    - bare 仓库与 linked worktree 拒绝打开，提示“该目录不是受支持的 Git 项目类型”（bare 无工作区；linked worktree 与内部临时 worktree 机制及单窗口锁模型冲突）。
  - 仓库没有 `origin` 远程时允许打开，即使存在其它 remote 也进入无远程模式（见“独特设计·远程仓库地址”）：主界面顶部显示警告条“未配置远程仓库”，同步/推送类入口隐藏，引导去项目设置填写远程地址。若存在其它 remote，额外提示本工具仅管理 origin。
- 右侧是之前打开过的项目列表，每个列表项左边是图标，然后经跟着图标的是左对齐的两行文本：目录名称和灰色字体的详细路径。点击可以打开项目。
  鼠标悬停在列表项上，会出现一排右对齐的按钮（仅图标）组，分别是：删除。
  列表的最底部有清空历史记录的文字链接形式的按钮。
  - 列表中的目录已被删除/移动时，点击提示，并提供“从列表中移除”按钮。
- 界面的右下角是右对齐的按钮（仅图标）组，从左到右是：向导、设置。

# 向导界面

两步式向导，右下角有“跳过”按钮（跳过则使用系统已有配置，之后可在设置中补填）；完成后进入起始界面。

- 第一步：作者信息
  - 要求用户输入 Git 使用的用户名和邮箱地址，这会存储起来，工具内部的操作均使用这些信息。
  - 自动预填系统 `~/.gitconfig` 中的现有值；邮箱做格式校验。
- 第二步：SSH Key
  - 检测用户是否有 Git 能使用的 SSH Key，如果没有则显示生成 SSH Key 按钮，并附上 SSH Key 作用说明。
    点击按钮会弹出窗口让用户输入信息以生成 SSH Key（尽量提供合理、常见的默认值让用户少输入）。
  - 检测规则：`~/.ssh/` 下存在任一常见私钥（id_ed25519/id_rsa/id_ecdsa）即视为“已有 Key”；生成按钮仅在完全没有时出现。
  - 生成弹窗：类型固定 `ed25519`、路径固定 `~/.ssh/id_ed25519`、注释自动用第一步的邮箱，均不给选项；唯一输入项为密码短语（passphrase），默认留空，附小字说明“留空则日常使用无需重复输入密码（推荐个人电脑使用）”。
  - 生成后、或已有 SSH Key 的情况下，显示复制 SSH Key（公钥内容）的按钮，并附上将 Key 添加到 Git 平台的小字说明文本，点击后复制到剪贴板。

- 向导可重入：起始界面右下角的“向导”入口允许用户随时重新走一遍（补身份、补 SSH Key），不受 onboarded 标志影响。
- 身份前置校验（懒校验）：在任何会创建 commit 或 stash 对象的操作（提交、撤回、含提交的自动跟踪同步，以及一切会自动/手动储藏的流程——带更改切换分支、带更改同步、进入审查模式、手动创建储藏）执行前，检查当前仓库有效身份（仓库级 → 全局 fallback）；实现收敛在写锁入口处：获取写锁后、执行前判断本次流程是否含 commit/stash 步骤。缺失则弹出“完善身份信息”弹窗（用户名 + 邮箱），填写后写入仓库级 config 并继续原操作，取消则中止。只读与纯引用操作（切换无更改分支、删除分支、Fetch）不校验。
- 邮箱一致性：身份里的邮箱同时用于历史图表的 Gravatar 头像匹配（仅在启用 Gravatar 时），一处配置多处复用。

# 主界面

- 选择项目后进入主界面。
- 主界面分为左侧栏和主面板区域。
- 左侧栏内容从上到下排列：
  - 项目信息区域，从左到右是图标；文本区域，文本区域是左对齐的两行文本：目录名称和灰色字体的详细路径；右对齐的一个同步按钮（仅图标），如果存在可同步的内容则会变成橙色，否则是正常主色。
  - 分支信息区域，列表的形式展示分支，每个列表项占一行，从左到右是：当前所在分支标记（点）、图标、分支名称，然后是右对齐的待同步提交数 Badge（如果没有待同步提交则不显示）。
    - 图标：本地分支显示为分支图标，远程分支显示为云分支图标。
    - 区域标题左对齐，同行右对齐是按钮（仅图标）组，分别是：新建分支。
    - 鼠标悬停在列表项上，会出现一排右对齐的按钮（仅图标）组，分别是：同步，切换，删除。
    - 点击分支会将主面板切换到历史选项卡，并聚焦到该分支的最新提交。
    - 右键列表项会弹出菜单，菜单内容为：同步，切换，作为基准创建新分支，删除。
  - 储藏信息区域，列表的形式展示储藏记录，每个列表项占一行，从左到右是：图标、储藏名称，然后是右对齐的储藏时间。
    - 区域标题左对齐，同行右对齐是按钮（仅图标）组，分别是：创建储藏。
    - 鼠标悬停在列表项上，会出现一排右对齐的按钮（仅图标）组，分别是：应用、删除、详情。
    - 创建储藏：弹窗输入名称（默认值“储藏于 <日期时间>”），并单选：储藏全部本地更改（默认）/ 仅储藏本地更改选项卡中勾选的文件；含未跟踪文件（`git stash push -u`），完成后工作区对应部分变干净。
    - 应用（手动）：`git stash apply`——保留储藏不删除（应用成功也不自动删）；冲突时进入冲突解决界面（文案：“储藏的内容 / 当前分支的内容”）。若用户在冲突界面取消，必须恢复到 apply 前状态，原储藏仍保留。
    - 删除：二次确认后 `git stash drop`。
    - 详情：弹出与提交详情面板同构的滑出面板——储藏名称、创建时间、文件列表 + Diff。
  - 分支信息和储藏信息区域均可以折叠，并有搜索框可以进行模糊搜索。
    - 未搜索到时会显示 “未搜索到相关内容” 的提示。
  - 审查模式按钮区域，有一个撑满该区域的青色渐变的 “进入审查模式” 按钮，点击后进入审查模式。
  - 更多操作按钮区域，从左到右排列的按钮（仅图标）组：设置。
  - 项目信息区域、审查模式按钮区域、更多操作按钮区域是固定高度的，分支信息和储藏信息区域本身的高度是自动撑满可用左侧栏高度的，并且有 handle 可以调节区域占用的尺寸，列表内容过多时会出现滚动条。
    调节的比例会被记录，下次打开项目时会恢复上次的比例。
- 主面板区域最上方有两个选项卡进行切换，默认显示 "历史" 选项卡，另一个是 "本地更改"。"本地更改" 选项卡标题带更改数 Badge（如 “本地更改 (12)”，为 0 时不显示数字）。
- 历史选项卡通过图表展示提交历史，图表上方有一行工具栏区域，分为左侧和右侧。
  - 图表形态：经典 commit graph——左侧彩色轨道线（分支线/合并线），每行一个提交：头像圆点、提交信息（单行截断）、分支/标签 Badge（指向该提交的分支头）、作者名、相对时间（悬停显示完整时间）。标签仅**只读展示**（图标/颜色与分支 Badge 区分开），工具不提供 tag 的创建/删除/推送。
  - 性能：虚拟滚动 + 分页加载（git log 按批 200 条），支撑万级提交仓库。
  - 头像来源：默认显示姓名首字母色块（纯本地生成）；仅当用户在设置中主动启用 Gravatar 时才联网取头像（取失败回退到首字母色块）。
  - 左侧为左对齐的工具组，分别是：分支筛选器。
    - 分支筛选器默认情况下是 “自动”，点击后会弹出下拉菜单，显示 “全部”、“自动” 两个特殊项，并有分割线分割，后跟所有分支项。
    - 每一项前面有复选框。
    - 当选择 “全部” 或者 “自动” 时，取消其它所有复选框的选择。
    - “全部”：即显示所有分支的提交历史。
    - “自动”：即显示当前所在分支的提交历史。
  - 右侧为右对齐的工具组，分别是：搜索框。
    - 搜索框默认情况下是空的，用户输入内容后会进行模糊搜索，搜索范围为提交信息、作者、提交内容。
    - 搜索实现：提交信息/作者用 `git log --grep/--author`；提交内容用 `git log -S`（pickaxe），结果合并去重，搜索期间显示 loading。
- 点击任何提交记录会弹出从下方滑出的提交详情面板，半透明遮罩阻挡界面下层点击，点击遮罩部分关闭面板。
  - 头部：完整提交信息、作者 + 头像、完整时间、短 hash（点击复制）。
  - 主体：该提交的变更文件列表（左） + 选中文件的 Diff（右），复用本地更改选项卡的 Diff 组件。
  - 操作按钮：“撤回此提交”、“复制提交号”。
- 本地更改选项卡显示当前所有未提交的更改，分为两个区域：
  - 左半部分是更改文件列表，列表的形式展示所有未提交的更改，每个列表项前面有个复选框，列表上方有一行工具栏，左对齐是一个复选框，有三个状态（全选、全不选、部分选），右对齐有一个搜索框。
    - 列表结构：默认平铺列表，每项显示文件名 + 灰色相对路径（过长省略，悬停看全路径）+ 变更类型图标（新增绿/修改黄/删除红/重命名蓝）；工具栏提供“平铺 / 目录树”切换按钮，目录树模式按文件夹层级折叠，记住上次选择的视图模式。
    - 重命名/移动检测：开启 git rename detection（`--find-renames`，二进制基于内容相似度），把“删除旧路径 + 新增新路径”识别为一条“重命名”记录，显示 `旧路径 → 新路径`；纯移动（内容不变）Diff 区提示“文件已移动，内容无变化”，移动+修改则展示实际差异。
    - 勾选粒度：目录树模式下勾选父文件夹 = 勾选其下所有更改文件（三态复选框），与平铺模式共享同一份勾选数据。
    - 列表项有两个状态组，一个是选中状态，一个是勾选状态。
    - 右键列表项会弹出菜单，菜单内容为：还原更改、储藏更改、勾选、取消勾选，针对的是所有选中的列表项。
      - 储藏更改：部分储藏，`git stash push -u -- <选中路径>`，弹窗输入名称（默认值同创建储藏）。
      - 还原更改：二次确认后丢弃选中文件的修改；所有丢弃类流程一致（含切换分支时选择丢弃），先把工作区当前版本按相对路径收集到临时目录并移入**系统回收站**作为隐性兜底，再恢复 Git 目标状态。红色警示文案保持“此操作无法撤销”（不承诺回收站）。不额外创建隐藏储藏备份（避免堆积与大二进制储藏成本）。
    - 搜索框：默认情况下是空的，用户输入内容后会进行模糊搜索，搜索范围为文件名和文件内容。
    - 列表下方是按钮组，右对齐的按钮组分别是：提交按钮（无勾选文件时禁用；提交信息在二次确认框中输入）。
    - 文件列表的勾选状态要尽量在内部操作中保持，比如自动储藏并恢复本地更改时，恢复后要保持原来的勾选状态。
  - 右半部分是 Diff 区域，显示当前选中更改文件的 Diff 内容。
    - 文本文件：标准 Diff 视图，支持“分栏 / 行内”两种模式切换（工具栏小按钮），语法高亮。
    - 图片文件（png/jpg/webp/gif/svg/bmp，含 LFS）：并排对比旧版本 | 新版本，显示尺寸与文件大小变化（如 `1024×1024 → 2048×2048，2.1MB → 4.3MB`）；新增文件只显示新图，删除文件只显示旧图；棋盘格背景（透明图友好），可缩放。
    - 其他二进制文件（PSD/模型/音频等）：不尝试渲染，显示文件卡片——图标 + 文件名 + 变更类型 + 大小变化（如 `12.4MB → 15.1MB`）。
    - LFS 指针文件：显示实际内容（图片走图片对比，其他走文件卡片），而不是指针文本；旧版本内容不在本地时按需 `git lfs fetch`，显示加载态。
    - 超大文本文件（>1MB 或 >5000 行变更）不渲染 Diff，显示“文件过大”卡片，避免卡死界面。
    - 以上 Diff 展示规则同时应用于：本地更改选项卡、提交详情面板、冲突解决界面。

# 冲突解决界面

呈现形式：全屏覆盖层（不是普通弹窗），顶部显示“正在解决冲突：<操作名>”，除完成/取消外无法离开（防止半冲突状态下误操作其他功能）；该状态下整个项目窗口的其他入口全部禁用。

该界面会展示所有有冲突的文件列表，并且展示选中文件的冲突内容。
需要标明哪个是他人的更改，哪个是自己的更改，比如软件内部的自动储藏时，储藏的内容称为自己的更改，当前分支的内容称为他人的更改。
但是手动应用储藏时，则直接称为储藏的内容，当前分支的内容。
该界面需要能够多选、全选、反选 + 使用任何一边的更改 + 进行手动编辑 + 保存更改。

- 布局：左侧冲突文件列表（每项复选框 + 已解决/未解决状态标记），右侧为选中文件的冲突详情。
- 顶部工具栏：全选/反选复选框 + 两个批量按钮“使用他人的更改”、“使用自己的更改”（作用于所有勾选文件，标签文案按场景变化，如“储藏的内容/当前分支的内容”）。
- 文本文件详情：左右双栏对比展示两边内容，操作：整文件选边按钮 + “手动编辑”模式。手动编辑用内嵌编辑器（CodeMirror）**预填 git 合并结果**：非冲突区域已自动合并，**冲突区域两边内容都保留、都可见**，以高亮装饰块 + 行内“采用他人这段/采用自己这段”小按钮呈现（不显示原始 `<<<<<<<` 标记，也绝不默认丢弃任何一边或静默拼接）；支持逐段选择与自由编辑并存。**保存门控**：仍有任何未决冲突区域时“完成/保存”禁用，保存时断言无残留未决区域、无残留 git 标记。
- 二进制文件详情：展示两侧版本的文件信息（大小、修改时间）；常见图片格式（png/jpg/webp/gif，含 LFS 文件先取回实际内容）显示缩略图预览对比；仅支持选边，不提供手动编辑。
- 完成/取消：所有文件已解决后“完成”按钮才可用，内部 `git add` 标记 resolved 并继续原流程（rebase/revert/stash 恢复）；“取消”按钮放弃整个操作并恢复到操作前状态，需二次确认。
- 选边实现：`git checkout --ours/--theirs -- <文件>`；rebase 场景下 ours/theirs 语义反转由内部正确映射，界面永远只呈现“他人的/自己的”这一层概念。

# 设置界面

弹出一个 Modal 窗口，左侧栏和右侧面板。

设置分为：

- 基本设置
  - Git 用户信息：用户名、邮箱地址。
  - SSH Key：显示当前使用的 SSH Key，并有复制按钮和生成按钮。
  - HTTPS 凭据：列表展示已保存的凭据（host + 用户名），支持修改和删除；高级项可展示与管理特定 path 的覆盖凭据。
  - 自动 Fetch 间隔：数字输入，单位秒，默认 60；校验范围 10–3600，越界红字提示并禁用保存。
  - 自动检查更新：开关（默认开启）；关闭后仍保留手动检查。
  - Gravatar 头像：开关（默认关闭）；开启后才联网获取头像。
  - 语言：跟随系统 / 中文 / English（默认跟随系统）。
  - 主题：跟随系统 / 浅色 / 深色（默认跟随系统）。
- 项目设置
  - 远程仓库地址：显示当前项目的远程仓库地址，并有复制按钮和修改按钮；允许清空（清空 = 删除 origin、仓库进入无远程模式，走二次确认）。
  - 忽略规则（.gitignore）编辑：提供编辑界面查看/修改仓库 .gitignore；属于仓库共享内容，保存后会作为一条本地更改待提交，需用户知情。
  - 大文件检查：开关（默认开启）+ 阈值（数字输入，单位 MB，默认 50）。
  - 自动跟踪设置：可以设置 A 分支追踪 B 分支，设置了自动追踪的 A 分支在执行同步操作时会使用不同的流程。
    - 这个设置是一个列表，有增删改按钮，点击后弹出一个 Modal 窗口让用户选择 A 分支和 B 分支。
    - 配置形态：每条规则 = (A 分支, B 分支)，A 唯一（一个分支只能追踪一个目标）；存储在项目级 projects.json。
    - Modal 内实时校验：A ≠ B（不能追踪自己）；禁止成环（图检测 A→B→…→A），违反则红字提示；A、B 均可选“仅远程”分支（按分支选择通用规则自动建本地跟踪分支）。
    - 失效处理：B 分支被删除 → 规则标记失效（列表中黄字“目标分支已删除”），A 同步时退化为普通同步（仅拉取推送 A 自身），不报错打断；A 分支被删除 → 规则自动移除。
    - 合并方向严格 ff-only：B 合入 A 若非 fast-forward（A 存在 B 没有的分叉提交）→ 提示“无法自动将 <B> 合并到 <A>，存在分歧提交”，然后进入恢复储藏流程，不自动 rebase（避免改写美术分支上可能已推送的提交）。
- 关于
  - 显示当前版本号、检查更新按钮、更新日志入口。
